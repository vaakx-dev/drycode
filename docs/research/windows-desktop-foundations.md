# Windows desktop host foundations

**Research ticket:** [docs: assess Windows desktop host foundations](https://github.com/vaakx-dev/drycode/issues/4)  
**Related map:** [docs: define Drycode's implementation-ready architecture](https://github.com/vaakx-dev/drycode/issues/1)

## Scope and reading

This assessment treats **Drycode Core** as the persistent desktop supervisor, not as the Shell Extension or Drycode Harness. **Reload** therefore means stopping and starting the extension and Harness runtimes while keeping the desktop host process and native window alive. The first release is Windows-only, extensions are fully trusted, and discovery is limited to the user-level `~/.drycode/extensions/` directory, as specified by the map.

The comparison is based on current first-party documentation from Electron, Tauri, and Microsoft. “Fact” means a documented capability or limitation. “Recommendation” means an architectural conclusion for Drycode, not a promise supplied by a framework.

## Short answer

**Electron is the lowest-risk fit for the current constraints.** Its main process is already a Node.js process with application-lifecycle and native APIs; it can create web windows, reload or replace their contents, and start Node-capable child processes. That makes a local TypeScript extension model and a separate Node-compatible Harness runtime straightforward. The cost is a larger bundled runtime and the need to design the trust boundary carefully: do not put replaceable extension code in the persistent main process.

**Tauri is a credible second choice** when installer size and use of the Windows WebView2 runtime outweigh implementation simplicity. Its Rust Core is a natural supervisor and its sidecar mechanism makes an out-of-process Harness possible. However, a TypeScript extension is not a native Tauri extension: it needs a bundled Node executable or packaged JavaScript sidecar, and frontend-to-native calls are deliberately permission/capability mediated. That is extra host contract and packaging work for a fully trusted, dynamically discovered extension directory.

**WinUI 3 + WebView2 is the credible first-party Windows alternative.** It offers the most direct Windows host and MSIX/App Installer story, but it supplies neither a Node runtime nor an extension/sidecar model. Drycode would have to build the supervisor, Node process management, JavaScript bridge, extension loading, and much of the update/install experience itself.

None of these foundations makes the exact Drycode Reload contract automatic. In every option, the host must own a lifecycle state machine that terminates and recreates the Shell Extension and Harness, handles stale IPC, and keeps the host window/supervisor alive.

## Concrete constraint matrix

| Constraint | Electron | Tauri | WinUI 3 + WebView2 |
|---|---|---|---|
| Persistent supervisor | Documented single Node main process owns app lifecycle and windows. | Documented Rust Core owns lifecycle, windows, state, and IPC. | Native WinUI process owns the window; WebView2 is an embedded, separately managed process group. |
| Replaceable web Shell Extension | `BrowserWindow` loads a local file or URL; `webContents` can navigate and reload. | Frontend runs in a system WebView; the Core can keep the window while the frontend is replaced/reloaded. | WebView2 can navigate/reload; the WinUI host can retain the native window and recreate the control. |
| Fully trusted local TypeScript | Node is built into main and UtilityProcess. A discovered extension can be loaded or started by host code. | TypeScript runs in the WebView or a Node sidecar; it does not become Rust Core code. | TypeScript runs in WebView2 or an independently shipped Node process; no Node runtime is supplied by WinUI. |
| Separate Node-compatible Harness | A dedicated child process is easy; Electron UtilityProcess is also Node-capable but is tied to Electron's runtime. | A Node sidecar is a documented pattern; long-lived IPC is custom. | A separately shipped Node process is custom but conventional for a native host. |
| Complete Reload without closing supervisor | Stop extension/Harness processes, recreate the renderer or load new Shell content, then re-establish IPC. | Stop/restart sidecars and recreate/reload WebView content while Rust Core remains alive. | Stop/restart child runtimes and close/recreate WebView2 controls while WinUI remains alive. |
| Local-only discovery | No framework catalog is required; main-process filesystem code can enumerate the chosen directory. | No framework catalog is required; Rust or a Node runtime can enumerate it, but frontend access is permission mediated. | No framework catalog is required; native host code can enumerate it. |
| Windows install/update | Electron `autoUpdater` supports MSIX and Squirrel.Windows, with packaging/tooling choices. | Tauri builds MSI or NSIS and its updater supports signed Windows artifacts. | Packaged MSIX supports the Windows App Installer/Store route; unpackaged WinUI has no App Installer automatic update. |
| Crash isolation | Renderer and UtilityProcess failures can be observed without putting the supervisor in the renderer. Main-process failure still ends the app. | WebView and sidecars are out of process; Core failure still ends the app. | WebView2 process failures are reported and recoverable by recreating/reloading controls; child-process recovery is host code. |
| Unrestricted native access | Main/child Node processes can use OS APIs and native Node modules under the user's Windows identity. | Rust Core has full OS access; frontend access is intentionally controlled by capabilities. A trusted Node sidecar can use the user's OS access. | Native host and child processes have normal Windows user/UAC access; web content needs an explicit bridge. |

The rows describe what the foundations provide. The extension directory, package format, IPC protocol, process topology, and Reload behavior remain Drycode design work in all three cases.

## Facts by foundation

### Electron

* Electron documents one **main process** per application. It runs in a Node.js environment, controls application lifecycle, creates `BrowserWindow` instances, and provides native APIs. Each window has a separate renderer process; destroying the window terminates its renderer ([Process Model](https://www.electronjs.org/docs/latest/tutorial/process-model)).
* A `BrowserWindow` can load a local HTML file or URL. `webContents` exposes `loadFile`, `loadURL`, `reload`, `render-process-gone`, and `unresponsive`; Electron documents reloading after forcefully replacing an unstable renderer ([BrowserWindow](https://www.electronjs.org/docs/latest/api/browser-window), [webContents](https://www.electronjs.org/docs/latest/api/web-contents)). This directly supports a replaceable Shell Extension and a host-preserving UI reload.
* Electron's renderer has no direct Node access by default. Preload scripts have Node access and can expose a deliberate bridge; `nodeIntegration` is disabled by default, `contextIsolation` is enabled by default, and enabling Node integration disables the renderer sandbox ([Process Model](https://www.electronjs.org/docs/latest/tutorial/process-model), [BrowserWindow](https://www.electronjs.org/docs/latest/api/browser-window)). “Fully trusted” removes the need for a security policy against an extension, but it does not require putting extension code in the supervisor. Keeping the bridge narrow still reduces accidental coupling and makes replacement possible.
* `utilityProcess.fork` creates a Chromium-managed child process with Node.js and message ports. It has `spawn`, `exit`, `error`, and `kill` events/methods and is documented for services that are CPU-intensive or crash-prone ([utilityProcess](https://www.electronjs.org/docs/latest/api/utility-process)). It is suitable for an extension runtime, but a Harness that must be independently versioned should instead be a separately shipped Node executable launched by the main process. The latter distinction is a Drycode requirement, not an Electron guarantee.
* Electron's Windows `autoUpdater` selects MSIX updating for MSIX packages and Squirrel.Windows for traditional packages. Its documented `quitAndInstall()` closes windows and quits the app to install the update ([autoUpdater](https://www.electronjs.org/docs/latest/api/auto-updater)). Thus app update is not Drycode Reload: an installer update may restart the supervisor, while Reload must not.
* Renderer process isolation is useful but not complete supervisor isolation: a renderer crash can be recovered, whereas a main-process crash takes down the application. A separate Harness and extension runtime should therefore not be loaded into main merely because extensions are trusted.

**Drycode implication:** Electron is unusually close to the desired TypeScript shape. The stable main process can discover `~/.drycode/extensions/`, own a persistent window, and manage a separately shipped Node Harness plus extension workers. The implementation must explicitly keep the main process small and define whether the Shell Extension is a renderer bundle, a local URL, or a new renderer/window contents instance.

### Tauri

* Tauri documents a **Core process** that owns full operating-system access, state, windows, and IPC, plus one or more WebView processes. On Windows the WebView is Microsoft Edge WebView2, which is dynamically supplied by the platform rather than bundled in the final executable ([Process Model](https://v2.tauri.app/concept/process-model/), [Prerequisites](https://v2.tauri.app/start/prerequisites/)). This maps well to a persistent native supervisor and a web Shell Extension.
* Tauri's frontend-to-Core API is not unrestricted by default. Permissions describe command privileges and scopes, and capabilities choose what a window/webview receives ([Permissions](https://v2.tauri.app/security/permissions/), [Capabilities](https://v2.tauri.app/security/capabilities/)). The docs also say that remote sources do not receive Tauri API access by default. Drycode can grant broad permissions because extensions are trusted, but must still design and maintain capability files or a Core command bridge. That is a material difference from “load arbitrary local TypeScript with Node.”
* Tauri explicitly documents external binaries (“sidecars”) and supports spawning them from Rust or JavaScript, with shell permissions and argument rules. Its Node guide packages a Node application as a self-contained Windows executable, or alternatively embeds the Node runtime and ships readable-ish JavaScript resources ([Embedding External Binaries](https://v2.tauri.app/develop/sidecar/), [Node.js as a sidecar](https://v2.tauri.app/learn/sidecar-nodejs/)). This makes an out-of-process Harness feasible, but means Drycode owns the Node packaging, protocol, and long-lived process lifecycle.
* Tauri's documented Windows installers are MSI (WiX) or NSIS setup executables. The installer can download, embed, or bundle WebView2, with corresponding internet, size, and offline-environment trade-offs ([Windows Installer](https://v2.tauri.app/distribute/windows-installer/)). Its updater supports signed Windows MSI and NSIS artifacts and requires a public/private signing key; on Windows it automatically quits the application before installing an update ([Updater](https://tauri.app/plugin/updater/)). Again, installer updates are distinct from Reload.
* Tauri describes asynchronous message passing between Core and WebView and JSON-serializable command arguments/results ([Inter-Process Communication](https://v2.tauri.app/concept/inter-process-communication/)). That is a sound basis for Shell/Core IPC, but it is a contract Drycode must define. A dynamic package discovered at runtime is not automatically a compiled Tauri plugin; the practical design is a stable Core bridge plus a Node extension/Harness process.

**Drycode implication:** Tauri can satisfy the requirements if Drycode treats Rust Core as the supervisor, runs trusted TypeScript outside the WebView in Node processes, and accepts a broad but explicit capability/IPC configuration. It is less direct than Electron for local TypeScript and dynamically discovered extensions, and its WebView2 runtime becomes an installation and compatibility decision.

### WinUI 3 + WebView2

* Microsoft documents a WinUI 3 desktop template using the Windows App SDK, which includes the WebView2 SDK. A WebView2 control can navigate to a URL, receive injected JavaScript, and be hosted by a native WinUI window ([Get started with WebView2 in WinUI 3](https://learn.microsoft.com/en-us/microsoft-edge/webview2/get-started/winui)). This supplies the native supervisor and replaceable web UI surface, but not Drycode's extension or Node runtime model.
* WebView2 uses a browser process, renderer processes, and helper processes. Microsoft exposes `CoreWebView2.ProcessFailed` and `BrowserProcessExited`; after a browser-process failure, controls must be recreated, and after a main-frame renderer failure, the host can call `Reload` or recreate controls ([Process model for WebView2 apps](https://learn.microsoft.com/en-us/microsoft-edge/webview2/concepts/process-model), [Handling process-related events](https://learn.microsoft.com/en-us/microsoft-edge/webview2/concepts/process-related-events)). This is strong evidence that a native host can preserve its window while recovering the Shell Extension, but Drycode must implement the recovery state machine.
* WinUI 3 does not provide Node.js or TypeScript extension loading. The native host would need to ship and start a Node-compatible Harness/extension executable and define a bridge between that process, WebView2, and the supervisor. That is a recommendation based on the requirement and the documented WebView2/WinUI role, not a framework feature.
* Microsoft documents both framework-dependent and self-contained Windows App SDK deployment. Packaged and unpackaged modes have different runtime and servicing consequences ([Windows App SDK deployment overview](https://learn.microsoft.com/en-us/windows/apps/package-and-deploy/deploy-overview)). In particular, Microsoft documents that unpackaged WinUI 3 apps have no automatic updates via App Installer or the Store, while packaged MSIX apps follow the package-identity deployment path ([Distribute an unpackaged WinUI 3 app](https://learn.microsoft.com/en-us/windows/apps/package-and-deploy/unpackage-winui-app), [Packaging overview](https://learn.microsoft.com/en-us/windows/apps/package-and-deploy/packaging/)).
* Microsoft documents App Installer update settings such as on-launch and automatic-background checks for apps installed from an `.appinstaller` file ([Configure update settings in the App Installer file](https://learn.microsoft.com/en-us/windows/msix/app-installer/update-settings), [Auto-update and repair apps](https://learn.microsoft.com/en-us/windows/msix/app-installer/auto-update-and-repair--overview)). This is the strongest first-party Windows update story, but it implies choosing packaged MSIX and settling package identity, signing, distribution, and update policy.

**Drycode implication:** WinUI 3 is viable if Windows-native UX, MSIX, and App Installer are more important than sharing the host's language/runtime with extensions. It has the greatest bespoke implementation surface: process supervision, Node bundling, local discovery, bridge APIs, and the exact installer/update workflow are all Drycode-owned.

## Recommendation (not yet a human decision)

1. **Prototype the architecture with Electron first.** Keep the main process as the persistent supervisor; load the active Shell Extension in a renderer; run each replaceable extension runtime and the separate Drycode Harness out of process; and implement Reload as an ordered stop, dispose, recreate, and reconnect operation. Use a standalone Node executable for the Harness if “separate” means separately versioned from Electron.
2. **Keep Tauri as the size/native-runtime alternative.** Select it if the team is willing to own Rust Core contracts, Node sidecar packaging, WebView2 provisioning, and broad trusted capabilities. It is not disqualified by the trusted-extension rule; it simply does not make that rule the default execution model.
3. **Choose WinUI 3 + WebView2 only with an explicit Windows-first decision.** It is attractive when MSIX/App Installer and native Windows integration outweigh the cost of implementing a complete Drycode host in C#/C++/WinRT plus a Node runtime. Prefer packaged MSIX if first-party automatic update is a requirement; do not assume an unpackaged installer supplies it.

This is a fit recommendation, not an architecture decision. A short proof-of-concept should verify: a discovered Shell Extension can be replaced; a crashing extension or Harness does not kill the supervisor; Reload leaves the native window alive; a standalone Node Harness reconnects; and the chosen Windows updater does not get confused with Reload.

## Assumptions and questions for the later human decision

* **Meaning of “separate Harness runtime”:** Does “separate” require a distinct process only, or a separately shipped/versioned Node runtime independent of Electron/Tauri/WinUI? This changes whether Electron UtilityProcess is acceptable.
* **Extension execution shape:** Is a trusted TypeScript extension source transpiled before discovery, loaded as JavaScript, or run through a bundled Node loader? Which Node version and native-module ABI must be supported?
* **Crash boundary:** Must every extension get its own process, or may multiple extensions share one extension runtime? The answer controls how much isolation Reload and recovery provide.
* **Shell bridge:** May the Shell Extension call unrestricted native operations directly, or must even trusted UI code use a stable Core IPC bridge? Tauri's capabilities make this explicit; Electron and WebView2 still need an intentional bridge to avoid coupling the Shell to host internals.
* **WebView2 policy:** Is the supported Windows baseline guaranteed to have Evergreen WebView2? If not, should the installer download a bootstrapper, embed an offline installer, or ship a fixed runtime? Tauri documents these as materially different size/offline choices; the same runtime question applies to a direct WebView2 host.
* **Installer policy:** Should the first release use MSIX/App Installer, MSI, NSIS, or a traditional installer? Does distribution require Store/package identity, per-user installation without elevation, offline installation, code signing, or enterprise deployment?
* **Update semantics:** May a product update close/restart the supervisor, as Electron and Tauri document for Windows updates, while Reload must not? If not, the update mechanism needs a separate bootstrapper/updater process and a stronger handoff design.
* **Trust and recovery:** “Fully trusted” rules out sandboxing and permission prompts as product policy, but does not answer whether a malformed extension should be prevented from taking down the supervisor. Should trusted code still be isolated for reliability, and what state must survive its restart?
* **Local discovery and writes:** Is `~/.drycode/extensions/` a fixed Windows path convention (resolved through the user profile), and how should partial installs, version collisions, and an extension changing while loaded be handled?
