# Pi source reuse obligations

## Conclusion

Drycode can use Pi source as the starting point for an independent Drycode Harness under the upstream repository's MIT license. The license expressly permits use, copying, modification, publication, distribution, sublicensing, and sale, and it does not impose a copyleft, source-publication, or upstream-synchronization condition. The operative condition is preservation of the upstream copyright and permission notice in copies or substantial portions of the software: [Pi `LICENSE`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/LICENSE).

That permission is about software copyright, not a clearance or license for the Pi name, logo, domain, package identifiers, or other branding. Pi's first-party materials consistently identify the project as “Pi” / “Pi Agent Harness,” and the CLI package publishes the `pi` executable: [repository `README.md`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/README.md), [coding-agent `README.md`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/README.md), [coding-agent `package.json`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/package.json). Drycode should therefore make the independent identity explicit and use Drycode naming throughout, subject to counsel's trademark and unfair-confusion review.

## Snapshot and scope

This review is pinned to upstream commit [`3da591ab74ab9ab407e72ed882600b2c851fae21`](https://github.com/earendil-works/pi/commit/3da591ab74ab9ab407e72ed882600b2c851fae21), the canonical `earendil-works/pi` repository (formerly reachable as `badlogic/pi-mono`). It covers the repository license, package manifests, first-party documentation, source headers, and vendored source visible at that commit. A future import must repeat the inventory against the exact commit(s) actually copied; this document is not a license audit of Pi's npm dependencies or of every asset in a resulting Drycode distribution.

## Copyright and license facts

- The repository has a root `LICENSE` file containing the MIT License and the notice `Copyright (c) 2025 Mario Zechner`: [LICENSE](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/LICENSE). The root package is private and has no `license` field, while the distributable package manifests declare `"license": "MIT"`: [agent](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/agent/package.json), [AI](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/ai/package.json), [coding agent](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/package.json), [TUI](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/tui/package.json), and [orchestrator](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/orchestrator/package.json).
- The MIT grant says recipients may deal in the software “without restriction,” including to use, copy, modify, merge, publish, distribute, sublicense, and sell it. It also permits those recipients to whom the software is furnished to do the same: [LICENSE, grant](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/LICENSE).
- The only express MIT condition in that file is that the copyright notice and permission notice be included in all copies or substantial portions. Drycode should keep the complete upstream `LICENSE` text in its source and distribution notices, and must not remove the Mario Zechner notice when replacing Pi implementation with Drycode-owned modifications: [LICENSE, condition](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/LICENSE).
- The license text contains no requirement to publish corresponding source, license modifications under MIT, contribute changes upstream, preserve Pi behavior, or maintain Pi compatibility. Those are characteristics of this permissive license text, not a promise that every file in the repository or every dependency is covered by it: [LICENSE](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/LICENSE), [root package metadata](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/package.json).
- The warranty and liability disclaimer must remain with the license notice; Drycode should not represent upstream Pi or Mario Zechner as warranting Drycode: [LICENSE, disclaimer](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/LICENSE).

The notice names Mario Zechner, while package metadata also lists authors such as Mario Zechner and Earendil Works. Those are the exact upstream records, but they should not be treated as a complete independent determination of contributor copyright ownership: [coding-agent metadata](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/package.json), [orchestrator metadata](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/orchestrator/package.json). Counsel should review provenance if Drycode copies material added by contributors or combines it with code from outside this repository.

## Source-header and third-party obligations

The upstream source does not put a Pi copyright header on every implementation file; for example, these first-party files begin with ordinary implementation comments/imports: [agent `src/index.ts`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/agent/src/index.ts) and [coding-agent `src/cli.ts`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/src/cli.ts). That absence does not remove the root MIT notice obligation for copied substantial portions.

Several files explicitly identify code derived from other projects. They are not safely covered by treating the root Pi license as the sole notice source:

- [`packages/coding-agent/src/utils/ansi.ts`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/src/utils/ansi.ts) identifies portions derived from `ansi-regex` and `strip-ansi`, and carries Sindre Sorhus's MIT copyright and permission text. Preserve that header and its links if the file or a substantial portion is copied.
- [`packages/tui/src/stdin-buffer.ts`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/tui/src/stdin-buffer.ts) says it is based on OpenTUI and identifies an MIT copyright. Preserve the attribution and obtain/check the corresponding OpenTUI license text before redistribution; the short source comment is not a substitute for an independent third-party notice inventory.
- [`packages/coding-agent/src/core/export-html/vendor/marked.min.js`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/src/core/export-html/vendor/marked.min.js) carries MarkedJS and Christopher Jeffrey copyright lines and identifies MIT licensing. Preserve those lines and verify the applicable Marked license when copying the vendored file.

Accordingly, the implementation import should maintain a provenance/notice ledger for copied source, generated output, vendored files, images, and dependencies. This is a compliance safeguard, not an assertion that all such material has the same license as Pi; the upstream repository itself contains the relevant source attribution in the files above and declares package dependencies separately in its manifests, for example [coding-agent `package.json`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/package.json).

## Naming and trademark boundary

- First-party documentation calls the project “Pi Agent Harness” and describes it as the home of Pi's self-extensible coding agent: [repository `README.md`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/README.md).
- The coding-agent documentation calls Pi a “minimal terminal coding harness,” uses Pi as the product name throughout, and documents Pi packages: [coding-agent `README.md`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/README.md).
- Upstream package metadata uses the `@earendil-works/pi-*` namespace, repository identity `earendil-works/pi`, and the `pi` binary: [coding-agent `package.json`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/package.json). The README also says the `pi.dev` domain was donated by exe.dev: [repository `README.md`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/README.md).
- The inspected upstream tree exposes a root software license but no separate trademark policy or trademark permission in the license text: [repository tree at the reviewed commit](https://github.com/earendil-works/pi/tree/3da591ab74ab9ab407e72ed882600b2c851fae21), [LICENSE](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/LICENSE). This is an observation about the reviewed materials, not proof that no trademark rights exist.

Therefore, Drycode should not retain `Pi` as its product, executable, package scope, logo, website/domain, or implied compatibility identity merely because it copied MIT-licensed code. It should use the established independent terms “Drycode Harness,” “Drycode Core,” and “Drycode Extension” from [Drycode's context](https://github.com/vaakx-dev/drycode/blob/82bde03183642f991a2b54d7e691441bad17ead9/CONTEXT.md). Whether a particular nominative reference such as “based on Pi source” is appropriate, and whether any Pi or third-party mark is registered or enforceable in a target jurisdiction, requires trademark counsel; the upstream MIT license does not answer those questions.

## Drycode implementation checklist

1. Record the exact upstream commit(s) copied and retain the complete upstream MIT `LICENSE` with the Drycode distribution: [upstream `LICENSE`](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/LICENSE).
2. Preserve all copied upstream and third-party source headers, including the three identified above; add Drycode's own copyright and a clear modification/provenance note without deleting upstream notices: [ANSI header](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/src/utils/ansi.ts), [OpenTUI header](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/tui/src/stdin-buffer.ts), [Marked header](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/src/core/export-html/vendor/marked.min.js).
3. Generate a Drycode third-party notices inventory for vendored code, assets, and dependencies rather than assuming package-level `MIT` metadata covers their contents: [coding-agent dependencies](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/package.json).
4. Rename user-visible and distribution identifiers to Drycode terms, remove Pi branding and Pi-specific paths/telemetry defaults unless deliberately retained for a documented compatibility reason, and avoid any appearance of upstream endorsement: [Pi package identity](https://github.com/earendil-works/pi/blob/3da591ab74ab9ab407e72ed882600b2c851fae21/packages/coding-agent/package.json), [Drycode naming context](https://github.com/vaakx-dev/drycode/blob/82bde03183642f991a2b54d7e691441bad17ead9/CONTEXT.md).
5. Before release, have counsel review the contributor provenance, third-party notice inventory, final branding, and any claim that Drycode is “based on” Pi. The upstream files establish the software-license facts above; they do not constitute a trademark clearance or jurisdiction-specific legal opinion.

## Legal-fact versus counsel boundary

The license permissions, notice condition, disclaimer, package declarations, source headers, and first-party names above are facts stated by the cited upstream files. The conclusions about preserving notices and avoiding a confusing brand follow from those facts and should be implemented as the conservative engineering posture. Trademark clearance, contributor ownership beyond the supplied notice, treatment of every dependency/asset, and jurisdiction-specific questions are cautions for counsel—not conclusions established by the MIT file.
